# goods_shop_design_funcoding

굿즈샵의 모든 화면이 "한 사람이 만든 것처럼" 보이게 하는 규칙이다.
새 페이지를 만들거나 기존 페이지를 고칠 때 이 문서를 기준으로 삼는다.

핵심 원칙 한 줄: **새 색·새 버튼 모양·새 크기를 즉흥적으로 만들지 않는다.**
필요한 건 대부분 이미 `assets/style.css`에 있다. 없으면 여기 규칙대로 추가한다.

---

## 기준 예시 — 이대로 만들면 됩니다

모든 페이지의 버튼과 입력칸은 아래 예시를 기준으로 삼는다.

### 좋은 버튼 예시

- 손가락으로 편하게 누를 만큼 크게 → `.btn`이 `min-height: 44px`로 보장한다
- 모서리는 부드럽게 둥근 모양 → `border-radius: 10px` (이미 `.btn`에 들어있다)
- 색은 화면에서 가장 잘 보이게 → 강조색 `--accent` 배경 + 흰 글자

```html
<button type="button" class="btn">결제하기</button>
```

새로 스타일을 짜지 말고 이 한 줄을 쓴다. 종류만 클래스로 바꾼다:
`btn-ghost`(보조·테두리만), `btn-danger`(삭제·빨강), `btn-block`(전체 너비).

### 좋은 입력칸 예시

- 입력칸 위에 무엇을 적는 칸인지 라벨을 표시한다 → `<label>`을 입력칸 바로 위에 둔다
- 잘못 입력하면 빨간색으로 알려준다 → 안내 문구에 `hint err` 클래스를 준다 (빨간 글씨)

```html
<div class="field">
  <label for="email">이메일</label>
  <input type="email" id="email" placeholder="you@example.com" required>
  <div class="hint" id="emailHint"></div>   <!-- 평소엔 비어 있음 -->
</div>
```

```js
// 잘못 입력했을 때: 빨간색으로 안내
emailHint.className = 'hint err';
emailHint.textContent = '이메일 형식이 올바르지 않습니다.';

// 올바를 때: 초록색으로 안내 (선택)
emailHint.className = 'hint ok';
emailHint.textContent = '사용할 수 있는 이메일입니다.';
```

`label` 없이 placeholder만 쓰지 않는다 — placeholder는 입력하면 사라져서 무슨 칸이었는지 잊게 된다. 라벨은 항상 남는다.

이 두 예시가 이 스킬의 "기준"이다. 새 페이지의 버튼·입력칸이 이와 다르게 생겼다면 맞춰서 고친다.

## 0. 먼저 할 일

페이지를 만들기 전에 `assets/style.css`를 연다. 거기에 색·버튼·카드·폼·배지·상태 스타일이 전부 정의돼 있다. 새 스타일을 인라인으로 쓰지 말고, 있는 클래스를 재사용한다.

모든 페이지의 공통 뼈대(복사해서 시작):

```html
<!DOCTYPE html>
<html lang="ko">
<head>
<meta charset="UTF-8">
<meta name="viewport" content="width=device-width, initial-scale=1">
<title>굿즈샵 — 페이지이름</title>
<link rel="stylesheet" href="assets/style.css">
</head>
<body>
<header class="site">
  <div class="wrap">
    <a class="brand" href="index.html">굿즈<span>샵</span></a>
    <nav class="site" id="authBox"></nav>
  </div>
</header>

<section>
  <div class="wrap">
    <!-- 내용 -->
  </div>
</section>

<footer class="site">
  <div class="wrap">
    <span>굿즈샵 · 토스페이먼츠 테스트 모드</span>
    <span>실제 결제가 아닙니다</span>
  </div>
</footer>

<script src="https://cdn.jsdelivr.net/npm/@supabase/supabase-js@2"></script>
<script src="assets/config.js"></script>
<script src="assets/format.js"></script>
<script src="assets/auth.js"></script>
<!-- 페이지 스크립트 -->
</body>
</html>
```

스크립트 순서(supabase CDN → config → format → auth)는 바꾸지 않는다.

## 1. 색 — 모든 페이지에서 같은 색

**색상값(#0d8f7f 같은 것)을 HTML이나 인라인 style에 직접 쓰지 않는다.**
색은 전부 `:root`의 CSS 변수를 쓴다. 새 색이 필요하면 인라인이 아니라 `style.css`의 `:root`와 다크모드 블록 **양쪽에** 변수를 추가한다.

| 쓸 곳 | 변수 |
|---|---|
| 배경 | `--bg`, `--bg-soft`, `--bg-card` |
| 글자 | `--text`(본문), `--text-dim`(설명·부가정보) |
| 강조·버튼·링크 | `--accent` |
| 성공/완료 | `--ok` | 
| 경고/대기 | `--warn` |
| 오류/삭제/실패 | `--danger` |
| 테두리·구분선 | `--line` |

**다크모드는 공짜가 아니다.** 새 변수를 넣으면 `@media (prefers-color-scheme: dark)` 블록에도 같은 이름으로 어두운 값을 넣는다. 라이트/다크 양쪽에서 다 확인한다.

---

## 2. 버튼 — 같은 모양, 손가락으로 누르기 쉽게

버튼은 항상 `.btn` 클래스로 만든다. `<a>`든 `<button>`이든 똑같이 쓸 수 있다.

```html
<button type="button" class="btn">결제하기</button>       <!-- 기본(강조) -->
<a class="btn btn-ghost" href="index.html">뒤로</a>        <!-- 보조(테두리만) -->
<button type="button" class="btn btn-danger">삭제</button>  <!-- 위험(빨강) -->
<button type="button" class="btn btn-block">전체 너비</button>
```

**손가락 터치 기준: 버튼 높이는 최소 44px.**
사람 손가락 끝이 눌리는 최소 크기가 그 정도다. 그보다 작으면 옆 버튼을 잘못 누른다.

- 기본 `.btn`은 이 기준을 만족한다. 모바일 주요 동작(결제·로그인·가입)에 이걸 쓴다.
- `.btn-sm`(작은 버튼)은 **높이가 44px보다 작으므로**, 데스크톱의 부차적 동작에만 쓴다. 모바일에서 손가락으로 누르는 주요 버튼에는 쓰지 않는다.
- 버튼끼리는 최소 8px 이상 띄운다(오터치 방지).

버튼 안 글자는 항상 있는 스타일 그대로 둔다. hover는 밝기+살짝 뜨는 효과가 이미 들어있으니 새로 만들지 않는다. 비활성은 `disabled` 속성만 주면 흐려진다.

---

## 3. 글씨 — 폰에서도 잘 보이게

- **본문 글씨는 16px(1rem) 아래로 내리지 않는다.** 그보다 작으면 폰에서 읽기 힘들고, 모바일 브라우저가 입력창을 확대해버린다.
- 설명·날짜 같은 부가정보도 **0.8rem(약 13px) 밑으로는 안 내려간다.**
- 줄 간격은 본문 `line-height: 1.7`을 유지한다(한글은 라틴 문자보다 넉넉한 줄 간격이 필요하다).
- 제목은 `h1`/`h2`/`h3`를 쓰고 크기를 인라인으로 바꾸지 않는다. 눈에 띄게 하려면 `.eyebrow`(작은 라벨) + `h1` 조합을 쓴다.
- 입력창 글씨도 16px 이상으로 둔다(이미 그렇게 돼 있다). 이 값을 줄이면 iPhone에서 입력 시 화면이 확대된다.

## 4. 모바일 반응형

`style.css`에 이미 두 개의 분기점이 있다. 새 분기점을 함부로 만들지 말고 이 둘을 기준으로 생각한다.

| 폭 | 무슨 일이 일어나는가 |
|---|---|
| 880px 이하 | 상품 3열 → 2열, 주문 행이 접힌다 |
| 560px 이하 | 상품 2열 → 1열, 헤더가 세로로 쌓인다 |

지킬 것:
- **가로 스크롤이 생기면 안 된다.** 넓은 표·이미지·코드가 있으면 그것만 `overflow-x: auto` 상자에 담고, 페이지 몸통은 절대 가로로 안 밀리게 한다.
- 표(`<table>`) 대신 **행 그리드**(`.orders`/`.order`)를 쓴다. 표는 좁은 화면에서 삐져나가지만, 행 그리드는 세로로 접힌다. 결제내역·관리자 목록이 이 방식이다.
- 크기는 `px` 고정보다 상대 단위를 쓴다. 너비는 `max-width: 100%`.
- 실제로 **폰 크기로 창을 줄여서** 확인한다. "아마 되겠지"로 넘기지 않는다.

## 5. 같은 의미는 같은 표현으로

같은 행동을 페이지마다 다른 말로 부르면 사용자가 헷갈린다. 아래 표현으로 통일한다. 새 문구가 필요하면 이 표에 추가하고, 이미 있는 건 그대로 쓴다.

| 상황 | 통일된 표현 |
|---|---|
| 삭제 재확인 | **"삭제할까요?"** (O) — "지우시겠습니까?"/"삭제하시겠어요?" (X) |
| 결제 완료 상태 | **"결제완료"** |
| 결제 대기 상태 | **"결제대기"** |
| 결제 실패 상태 | **"결제실패"** |
| 결제 버튼 | **"결제하기"** |
| 로그인 필요 안내 | **"로그인이 필요합니다."** |
| 저장/제출 완료 | **"저장했습니다."** |
| 되돌아가기 | **"뒤로"** |
| 금액 표기 | `Fmt.won()` 사용 → **"12,000원"** (쉼표 + "원", 콤마 없는 raw 숫자 금지) |
| 날짜 표기 | `Fmt.date()` 사용 (직접 포맷하지 않는다) |
| 상태 배지 | `Fmt.statusBadge()` 사용 (색·글자를 손으로 만들지 않는다) |

상태 배지·금액·날짜는 **반드시 `format.js`의 `Fmt` 헬퍼를 거친다.** 페이지마다 직접 만들면 표기가 갈라진다.

문장 톤: 존댓말, 간결하게. 느낌표를 남발하지 않는다("완료됐습니다" O / "완료됐어요!!" X).

## 6. 하지 말 것 (금지 규칙)

새로 만드는 페이지는 아래를 지킨다. 이건 "되도록"이 아니라 금지다.

1. **이모지·아이콘을 새로 넣지 않는다 — 글자로만 표시한다.**
   - 상태·안내·버튼 라벨에 이모지를 붙이지 않는다. `"✅ 결제완료"` (X) → `"결제완료"` (O)
   - 장식용 아이콘(✅ ⚠️ 😕 📦 🔒 등)을 새 페이지에 넣지 않는다. 필요한 뜻은 글자로 적는다.
   - **예외 — 상품 이미지:** `products` 테이블의 `emoji` 값은 상품 사진을 대신하는 것이라 그대로 쓴다. 이건 아이콘이 아니라 상품 그림이다.
   - 기존 6개 페이지(index/login/orders/admin/payment-*)의 이모지는 이 규칙 이전에 만든 것이라 건드리지 않는다. **새 페이지부터** 적용한다.

2. **페이지마다 다른 색을 쓰지 않는다 — 정해진 색만 쓴다.**
   - 색은 `style.css`의 `:root` 변수(`--bg`·`--text`·`--accent`·`--ok`·`--warn`·`--danger`·`--line`)에서만 고른다.
   - 새 색을 hex(`#...`)로 만들지 않는다. HTML·인라인 style에 색상값을 직접 쓰지 않는다.
   - 강조색은 항상 `--accent` 하나로 통일한다. 페이지마다 포인트 색을 바꾸지 않는다.

3. **같은 의미를 페이지마다 다른 단어로 쓰지 않는다.**
   - 한 가지 동작에는 한 가지 단어만. `"저장"` / `"확인"` / `"완료"`를 섞어 쓰지 않는다.
   - 5번 표의 통일 표현을 따른다. 표에 없는 새 단어가 필요하면 **먼저 표에 추가하고** 그 단어로 통일해서 쓴다.

## 7. 마지막 점검 (페이지를 끝내기 전에)

- [ ] 색을 인라인에 하드코딩한 곳이 없다 (전부 `var(--...)`)
- [ ] 버튼은 `.btn` 계열을 썼고, 모바일 주요 버튼은 44px 이상이다
- [ ] 본문 16px 이상, 부가정보 13px 이상
- [ ] 폰 폭(≤560px)에서 가로 스크롤이 안 생긴다
- [ ] 금액·날짜·상태는 `Fmt` 헬퍼를 거쳤다
- [ ] 버튼·상태 문구가 5번 표와 일치한다
- [ ] 다크모드에서도 글자가 배경에 묻히지 않는다
- [ ] `innerHTML`에 넣은 사람이 입력한 값은 `Fmt.escape()`를 통과했다
- [ ] (새 페이지) 이모지·장식 아이콘을 안 썼다 — 상품 이미지 이모지만 예외
- [ ] 강조색은 `--accent` 하나로 통일했고, 새 색을 만들지 않았다
- [ ] 같은 동작에 같은 단어를 썼다 (저장/확인/완료 섞지 않음)

이 스킬은 디자인 일관성만 다룬다. 보안·DB·결제 로직은 `CLAUDE.md`와 `ARCH.md`를 따른다.
